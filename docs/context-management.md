# Управление контекстом

---

## Почему контекст важен

Claude Code работает с ограниченным окном контекста. Каждое сообщение, каждый файл, каждый ответ — всё занимает токены. Когда окно заполняется, Claude Code либо начинает «забывать» старые части разговора, либо вовсе отказывается продолжать.

Это создаёт два типа проблем:

**Проблема 1: стоимость.** Каждый токен стоит денег. Длинная история разговора, которая многократно передаётся в API, умножает расходы. Реальный пример из логов (`compact-check.ts`):

> Данные за 01.06: сессия дошла до 203% из-за порога 75% и долгих задач.

При 203% заполненности cache_read токены составляли сотни миллионов — это многократная переотправка одних и тех же данных.

**Проблема 2: качество.** Переполненный контекст снижает качество ответов. Claude «разбавляется» лишней информацией и хуже фокусируется на текущей задаче.

### Пример расходов

При активной работе с кешированием типичная сессия может накапливать:
- **input_tokens:** ~10 000–50 000 за ход (новые данные)
- **cache_creation_input_tokens:** ~50 000–200 000 за сессию (создание кеша)
- **cache_read_input_tokens:** миллионы за длинный день (повторное чтение кеша)

Если не управлять контекстом, `cache_read` растёт экспоненциально по мере удлинения истории.

## Три команды управления контекстом

### `/compact` — сжатие с сохранением сути

Claude Code анализирует текущую историю разговора и создаёт её компактное резюме. Детали теряются, но ключевые факты и контекст задачи сохраняются.

**Что сохраняется:**
- Цель текущей задачи
- Принятые решения и их обоснование
- Текущее состояние работы
- Важные находки (ошибки, архитектурные решения)

**Что теряется:**
- Дословный текст предыдущих сообщений
- Промежуточные результаты которые уже учтены
- Подробности файлов которые уже обработаны

**Результат:** контекст уменьшается примерно с 35–50% до 10–15%. Разговор продолжается естественно.

### `/clear` — полный сброс

Удаляет всю историю. Claude «просыпается» с чистого листа.

**Что теряется:** всё. Вся история разговора, контекст задачи, результаты предыдущих операций.

**Когда имеет смысл:**
- Задача полностью завершена
- Нужно переключиться на совершенно другую задачу
- Контекст настолько переполнен что compact уже не поможет

### `/newsession` — новая сессия

Убивает screen-сессию и запускает новую. Ещё более радикально чем `/clear` — Claude Code перезапускается полностью.

**Отличие от `/clear`:** перезапускается весь процесс Claude Code, не только история. Все загруженные ресурсы, инструменты, соединения — пересоздаются.

**Время оффлайна:** ~15 секунд.

## Автоматическая двухуровневая стратегия

Система автоматически управляет контекстом через хук `compact-check.ts`, который запускается при каждом Stop-событии.

### Уровень 1: compact (COMPACT_THRESHOLD = 35%)

Когда заполненность контекста достигает 35%, хук:
1. Отправляет уведомление в Telegram: `🗜 Контекст 38% — запускаю /compact`
2. Инжектирует `/compact` в screen-сессию
3. Claude делает compact и продолжает работу

### Уровень 2: clear (CLEAR_THRESHOLD = 80%)

Когда заполненность достигает 80%, хук:
1. Отправляет уведомление: `🧹 Контекст 83% — критично, запускаю /clear (полный сброс)`
2. Инжектирует `/clear`

### Почему два уровня, а не один

**Проблема долгих задач:** Stop-хук срабатывает только когда Claude завершил ход. Длинная задача (сборка, тестирование, рефакторинг большого файла) может занять 10–20 минут без единого Stop-события. За это время контекст вырастает непрерывно.

**Пример:** установлен только один порог 35% (compact). Claude начинает большую задачу при 30% — к моменту завершения контекст уже 120%. Compact слишком поздно. Поэтому существует второй уровень — при 80% всё равно запустить хоть что-то (clear), пока не стало 100%+.

```
Начало задачи:    30% ──────────────────────────────────────────► 120%
                                                            Stop-хук
                                                         compact-check.ts
                                                      120% ≥ 80% → /clear
```

### Расчёт процента

Используется формула:

```
pct = (input_tokens + cache_creation_input_tokens) / context_window_tokens * 100
```

**Почему `cache_read` исключён:**

Многоуровневый кэш Claude может возвращать `cache_read_input_tokens` больше размера контекстного окна. Это нормально технически (кэш хранит много версий), но если использовать его для расчёта — получаем 150–200%, что ложно триггерит `/clear` каждый раз.

Правильная метрика — только «свежие» токены: те что добавились в текущем и предыдущих ходах (`input + cache_creation`).

## Настройка порогов

### Через переменные окружения

В `claude-bot.sh` и `launchd-boot.sh` читаются переменные:

```bash
COMPACT_THRESHOLD=35   # процент для compact
CLEAR_THRESHOLD=80     # процент для clear
```

### Через plist (постоянно)

Задать в `com.alexlevnikov.claude-buildserver.plist`:

```xml
<key>COMPACT_THRESHOLD</key>
<string>40</string>
<key>CLEAR_THRESHOLD</key>
<string>75</string>
```

После изменения перезагрузить launchd:

```bash
launchctl unload ~/Library/LaunchAgents/com.alexlevnikov.claude-buildserver.plist
launchctl load ~/Library/LaunchAgents/com.alexlevnikov.claude-buildserver.plist
```

### Рекомендуемые значения

| Сценарий | COMPACT_THRESHOLD | CLEAR_THRESHOLD |
|---|---|---|
| Обычная работа | 35 | 80 |
| Экономный режим | 25 | 60 |
| Агрессивный compact | 20 | 50 |
| Без автоматизации | 100 | 100 |

## Жизненный цикл сессии

```
Запуск сессии (0%)
        │
        ▼
  Разговор идёт, контекст растёт
        │
        ▼ (каждый Stop-хук)
  compact-check.ts проверяет %
        │
  ┌─────┴──────────────────────────────┐
  │                                    │
  │ < 35%                          ≥ 35%
  │ Ничего                          │
  ▼                                 ▼
  Продолжить                    ≥ 80% ?
                                  │
                     ┌────────────┴────────────┐
                     │                         │
                   35–79%                    ≥ 80%
                     │                         │
                     ▼                         ▼
                  /compact                  /clear
                  🗜 уведомление           🧹 уведомление
                     │                         │
                     ▼                         ▼
               Контекст ~10%           Контекст 0%
               (суть сохранена)        (полный сброс)
```

## Ручное управление

Помимо автоматики, можно управлять вручную через Telegram:

**Кнопка 🗜 Compact:**
```
Нажать кнопку → ответ «🗜 Запускаю /compact...»
→ Claude делает compact → контекст уменьшается
```

**Кнопка 🧹 Clear:**
```
Нажать кнопку → «Очистить контекст? Ответь 'да'»
→ написать 'да' → /clear выполняется
```

**Кнопка 🆕 Новая сессия:**
```
Нажать → «Создать новую сессию? Ответь 'да'»
→ 'да' → new-session.sh main → 15 сек оффлайн → новая сессия
```

## Советы по снижению стоимости

**1. Используйте compact регулярно.** Не ждите автоматики — если задача завершена, вручную нажмите 🗜 перед следующей.

**2. Разбивайте большие задачи.** Лучше несколько сессий по одной задаче, чем одна огромная. Каждый `/newsession` полностью сбрасывает счётчик токенов.

**3. Не держите одну сессию вечно.** Даже с compact история накапливается. `/newsession` раз в день — хорошая практика для интенсивной работы.

**4. Используйте Haiku для простых задач.** Через кнопку 🤖 Модель переключите на `claude-haiku-4-5` для чтения файлов, быстрых ответов, простых вопросов. Opus нужен только для сложной разработки.

**5. Compact агрессивнее при работе с большими файлами.** Снизьте `COMPACT_THRESHOLD` до 20–25% если часто работаете с файлами > 1000 строк.

## Мониторинг использования

Текущий процент заполненности можно увидеть в логах хука после каждого хода:

```bash
tail -f ~/.claude/buildserver/launchd.err.log
```

Или через `/logs` в Telegram — в выводе screen-сессии Claude Code периодически показывает использование контекста.

Расход TTS по дням:

```bash
cat ~/.claude/buildserver/state/tts-usage.json
```

```json
{
  "2026-06-01": 12340,
  "2026-06-02": 8900
}
```
